System Prompt Architecture

Prompt 結構解構與意圖分析

該 Prompt 旨在構建一個能夠維持「狀態」、執行「工具調用」並輸出「結構化 JSON」的自動化代理人。以下是其五大核心模組分析。

Module 1: 變數注入

動態上下文 (Jinja2 Style)

Prompt 使用雙大括號 {{variable}} 作為佔位符。這意味著 Prompt 本身是一個模板，後端程式會在發送給 LLM 之前動態填入數據。

  • {{agent_name}} // AI 名字
  • {{user_name}} // 用戶名字
  • {{activity_json}} // 當前任務狀態
  • {{context_data}} // RAG 檢索資料
Module 2: 角色定位

目標導向代理人

定義了 AI 的核心任務是「協助用戶達成目標」。

關鍵設計： 要求 AI 在每個 Session 開始時「自我介紹」並「列出可用功能」。這有助於讓用戶快速了解 AI 的邊界。

Module 3: 行為準則與護欄 (Guardrails)

規則解析

防幻覺 (Anti-Hallucination)

規則明確指出："Only suggest actions specifically defined in Artifacts.Utilities"。
AI 被禁止發明不存在的 API 參數或功能，這是 Tool Use 場景中最關鍵的安全設定。

思維鏈 (CoT)

要求在執行動作前提供 "high-level summary" 和 "concise checklist"。
這強制模型先進行規劃再執行，提高複雜任務的成功率。

錯誤處理

設定了明確的 Error Flags，例如 AMBIGUOUS_COMPLEXITY 或 SCHEMA_MISMATCH。這讓前端 UI 可以捕捉這些錯誤並顯示適當的引導，而不是讓 AI 胡亂回答。

用戶體驗優化

強制要求使用 Activity.Artifacts 中的 "Human-readable names" 而非內部 ID。這是為了確保對話自然，不會洩露後端資料庫結構。

Module 4: 輸出協議 (Output Protocol)

JSON Enforced Mode

此 Prompt 不允許 AI 輸出自由文本，而是強制要求 Single JSON Object。這通常是為了讓應用程式（App/Web）能夠直接解析回應。

  • message 給用戶看的 Markdown 訊息。
  • question 引導對話的下一個問題（Slot Filling）。
  • error 狀態檢查，若有缺漏參數則報錯。
// 預期輸出範例
{ "message": "I can help you setup the daily report.", "question": "What time would you like to receive it?", "summary": null, "error": null }
Module 5: 狀態範例 (Few-Shot Learning)

Activity Frame 狀態機

Prompt 中提供了一個巨大的 JSON 範例 (Activity, Motive, Object, Action_Sequence)。

設計意圖：

這不僅是範例，更是定義了 Agent 的「記憶結構」。AI 需要在對話過程中，將用戶的意圖（Needs/Interests）填入這個結構，最終生成一個完整的 Action_Sequence 供後端執行。這是一種 "Slot Filling"（槽位填充）的高級應用。

總結

這個 Prompt 是一個「生產環境級別」的指令集。它犧牲了 AI 的自由創造力，換取了極高的穩定性、可預測性與系統整合能力。它將 LLM 從一個「聊天機器人」轉變成了一個嚴格遵守業務邏輯的「後端控制器」。
